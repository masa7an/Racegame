## 1. 決定事項（ユーザー確認済み）

- **機能範囲**: 単発ギミック**背景**（見た目のみ）。物理挙動（グリップ・ブレーキ等）には影響させない。
  - Stage4（砂）・Stage5（ウェット）のような `car.py` 側の `if stage_id == N` 分岐は追加しない。
- **配置方式**: **新規専用ステージ（Stage6）**として追加する。中身は **Stage1の設定をコピー**したものをベースとし、コースの**一部区間**にトンネル演出を挿入する（Stage6全体をトンネルにはしない）。
  - `STAGE_CONFIG[6]` は `STAGE_CONFIG[1]`（環境色・カーブ特性・縁石あり等）を複製した内容にする。
  - ※ `create_road()` は `stage_id` を乱数シードに使っている（[track.py:170](../src/track.py:170) `rnd = random.Random(s_id)`）ため、設定値をStage1と同じにしても、Stage6は**Stage1と同じ配色・同じ難易度傾向だが、実際のカーブ順序はStage1とは異なる別コース**になる。「見た目・難易度の系統がStage1と同じ」という意味でのコピーであり、コースレイアウトの完全複製ではない点に注意（もし完全に同一レイアウトが必要な場合はシード固定の追加対応が必要）。

## 6. 断面形状を半楕円アーチに変更（2026-07-17）

直方体（垂直な壁2枚＋平らな天井1枚）から、円柱を横に半分に割ったような半楕円の弧に変更。
- 従来の `TUNNEL_HEIGHT`（アーチ頂点の高さ）・`TUNNEL_HALF_WIDTH`（路面との接点までの幅）はそのまま弧の縦横の半径として流用。幅と高さを別々に持つため厳密な真円ではなく半楕円（円柱っぽく見せる目的で幅・高さは独立調整可能な方を採用）。
- 弧は `TUNNEL_ARC_SEGMENTS = 8` 分割の台形ポリゴンで近似（`theta = 0〜π` を8分割し、`x = HALF_WIDTH*cos(theta)`, `y = HEIGHT*sin(theta)` で断面上の点を計算）。
- 面ごとの陰影（明暗の作り分け）は行わず、`TUNNEL_ARCH_COLOR` 単色（フォグのみ通常通り距離でブレンド）。旧 `TUNNEL_WALL_COLOR`/`TUNNEL_CEILING_COLOR` の2色使い分けは廃止し、`TUNNEL_ARCH_COLOR` に一本化。
- ヘッドレス描画で再確認済み。アーチの輪郭は8分割のファセットがわずかに見えるが、通常のプレイ距離では滑らかな半円状に見える。

## 7. 天井ライトの追加（2026-07-17）

### 7-1. 決定事項（要件整理）
- **見た目**: 単色の平面（天井上の明るい図形のみ）。グロー・にじみ演出は入れない。
- **配置間隔**: 密（短い間隔で連続的に見える）。
- **路面への影響**: 入れない（天井の装飾のみ、路面色・車体は変化しない）。

### 7-2. 実装内容
- 追加定数（[track.py](../src/track.py) 冒頭）: `TUNNEL_LIGHT_COLOR`（暖色系）、`TUNNEL_LIGHT_ARC_RANGE = (3, 5)`（アーチ頂点付近2ファセットのみ点灯対象）、`TUNNEL_LIGHT_SPACING = 4`（4セグメント周期）、`TUNNEL_LIGHT_ON_LENGTH = 2`（周期のうち2セグメント点灯＝密な配置）。
- 半楕円アーチのファセット描画ループ内で、`seg['index'] % TUNNEL_LIGHT_SPACING < TUNNEL_LIGHT_ON_LENGTH` の場合のみ、頂点付近のファセット色をアーチ色からライト色に切り替える。新規ジオメトリは追加せず、既存のアーチ面を塗り分けるだけなので実装コストが小さい。
- ライト色も通常のフォグ処理（`fog_pct`によるブレンド）を通すため、奥のライトほど暗闇に沈んで見える（近くのライトほど明るい）。
- ヘッドレス描画で確認：トンネル内部からは、頂点付近に山形のライトが等間隔に連なって奥へ消えていく見た目になっている。

### 7-3. 調整：左右2灯に分離・縮小（2026-07-17）
- 元の実装は弧のファセット（`TUNNEL_ARC_SEGMENTS`分割）をそのまま点灯色に塗り替える方式だったため、頂点を挟んだ左右のファセットが隣接し「1本の帯」に見えていた。
- 弧のファセット分割とは独立した角度でライト用の四角形を重ね描きする方式に変更（`tunnel_arc_pt(theta, ...)` ヘルパーを追加）。これにより弧の滑らかさ（8分割）を変えずにライトのサイズ・位置を自由に調整できる。
- 新定数: `TUNNEL_LIGHT_CENTER_OFFSET`（頂点からライト中心までの角度、20°）、`TUNNEL_LIGHT_HALF_ANGLE`（ライト自体の半幅、7°＝全体14°）。左右対称に2灯を配置し、中心に約26°分のギャップができる。
- サイズは旧実装（1本45°）に比べて1/3程度に縮小。
- ヘッドレス描画で再確認済み：左右にはっきり分かれた小さめのライトが奥へ連なる見た目になっている。

## 11. グローを縮小→拡大ブラー方式へ作り直し（2026-07-17）

10-②の方式（本体より広い角度の半透明ポリゴンを2段階重ねる）では、輪郭のはっきりした台形が重なるだけで
「オレンジのライトを色違いの黄色いパネルが囲んでいる」ようにしか見えず、発光感・にじみ感が出なかった。
角度を変えた多層ポリゴンには連続的な減衰がないことが原因なので、ブラーを実際にかける方式へ変更した。

- **ライト本体**: 変更なし。従来どおり台形ポリゴンを直接スクリーンへ描く（輪郭はシャープ）。
- **発光用Surface**: `self._tunnel_glow_surf` を `SRCALPHA` から**黒で塗り潰した不透明Surface**へ変更。加算合成では黒＝発光なしとして扱われるため、縮小時にライトの色が周囲の黒と混ざって減衰が生まれる。ここへは**ライト本体と同じ形のポリゴンだけ**を描く（広げたポリゴンは不要）。
  - 描画色は `interpolate_color(TUNNEL_LIGHT_COLOR, (0,0,0), fog_pct)`。奥ほど黒へ落ちるので、`glow_fade` のような明示的な減衰計算は不要になり削除した。
- **疑似ブラー** (`Track._blit_tunnel_glow`): 発光用Surfaceを `pygame.transform.smoothscale` で縮小→拡大する。拡大時の補間により、多層ポリゴンでは作れない連続的な減衰＝にじみが得られる。
  - 新定数 `TUNNEL_LIGHT_GLOW_LEVELS` = `((4, 1.0), (12, 0.7))`（`(縮小率, 強さ)` を縮小率の小さい順に並べる）。縮小率が大きいレイヤーほど広く弱いにじみになる。各レイヤーは**前段の縮小結果からさらに縮小**するので、一気に縮小するより滑らかで安い。
  - 強さの調整は `fill((w,w,w), BLEND_MULT)` による乗算で行う。加算合成ではアルファ値が無視されるため、`set_alpha` は使えない。
  - `TUNNEL_LIGHT_GLOW_INTENSITY`（既定 1.0 ＝ 素通し）は全レイヤーの強さに一律で掛かる倍率。レベルごとの比率を保ったままグロー全体を増減したい時だけ触る。
  - 各レイヤーを `smoothscale` で全画面へ拡大し、`BLEND_ADD` でスクリーンへ重ねる。強さ1.0のレイヤーが重なる中心は白飛びし、光源の芯として見える。
  - 縮小/拡大の中間Surfaceは `self._glow_scratch` にサイズ別でキャッシュし、毎フレームの確保を避ける。
- ヘッドレス描画（トンネル内部3地点）で確認済み：滲みが連続的に減衰し、近距離のライトは芯が白飛びして発光しているように見える。パネル状の見えは解消。
- 実測コストは約0.6ms/frame（トンネル内 7.4ms / トンネル外 6.8ms、800x600・ヘッドレス）。60fpsの予算16.6msに対して十分小さい。

## 12. 入口に厚み（小口面）を追加（2026-07-17）

アーチが薄い板に見えてトンネルとして不自然だったため、入口坑口に厚みの見える面を追加した。

- **構成**: 入口セグメント（`tunnel_start` をまたぐ最初のセグメント）の手前面上に、外枠アーチ（半径 `+TUNNEL_PORTAL_THICKNESS`）と内枠アーチ（＝坑内の弧と同じ半径）の二重ポリゴンを取り、間を `TUNNEL_ARC_SEGMENTS` 分割の四角形で埋める。既存の `tunnel_arc_pt()` に半径引数（`hw`/`h`）を追加して流用した。
- 新定数: `TUNNEL_PORTAL_THICKNESS = 170.0`（`ROAD_WORLD_WIDTH`=約10m換算で約50cm相当）、`TUNNEL_PORTAL_COLOR = (120, 118, 122)`（外光に照らされたコンクリート想定。坑内のアーチ色より明るくして厚みが読めるようにした）。
- 小口面は坑外（外光側）にあるため、フォグ到達色は他のトンネル要素と違い `TUNNEL_FOG_COLOR`（暗闇）ではなく**通常の `fog_color`（空色）**へブレンドする。周囲の昼光と馴染む。
- 描画順は坑内アーチ・天井ライトの後（同じセグメントのループ内）。半径が坑内の弧より外側なのでZファイティングは起きない。
- **奥行き方向へは延ばしていない**: 当初は「外アーチ→少し奥へオフセットした内アーチ→側面を四角形で接続」という押し出し方式で実装したが、内枠から奥の空間は坑内の弧そのものが既に埋めており冗長だった。加えて入口手前1,000ユニット以内ではその側面を極端な斜め（ほぼ真横）から見ることになり、画面の周辺全体が明るい小口色で覆われて「灰色の壁に突っ込む」見えになった。入口面上の平らなリング1枚に変更してこの破綻を解消（ヘッドレス描画で入口手前600ユニットの絵が変更前＝リングなしと完全一致することを確認）。
- ヘッドレス描画5地点（手前6,000／2,500／600／100ユニット・内部）で確認済み：接近時に開口部を縁取るコンクリートの帯として見え、遠方では細い明るい輪郭、至近距離・内部では破綻なし。

## 13. 断面を滑らかに（弧の分割数を距離から決める）（2026-07-17）

固定8分割では接近時にアーチの輪郭が多角形に見えていた（6節参照）。一律に分割数を上げると
トンネルが遠いフレームまで重くなるため、**フレームごとに必要な分割数を計算する**方式にした。

### 13-1. 実装
- `TUNNEL_ARC_SEGMENTS = 8`（固定値）を廃止し、`Track._arc_segments_for(radius_px)` を新設。
  N分割の内接多角形は弦の中央で真円より `sagitta = R*(1-cos(pi/2N))` だけ凹む。これが多角形に
  見える正体なので、凹みが許容値以下になる最小のNを小角近似 `sagitta ≈ R*(pi/2N)^2/2` から逆算する。
- 新定数: `TUNNEL_ARC_TOLERANCE_PX = 0.8`（凹みの許容量。小さいほど滑らかで重い）、
  `TUNNEL_ARC_SEGMENTS_MIN = 4`、`TUNNEL_ARC_SEGMENTS_MAX = 24`。
- 基準半径は**最寄りのトンネル断面**の投影後の半径（＝画面上で最も大きく写り、多角形が最も目立つ位置）。
  結果の `tunnel_arc_n` を描画ループの前で1回だけ求め、全トンネルセグメントで共有する。
- `MAX = 24` の根拠: 接近時（入口手前2,500）に必要なのは22分割なので画質には影響せず、
  弧が画面外まで広がる至近距離のコストだけを抑える。

### 13-2. 重要: 分割数はセグメントごとに変えてはいけない
最初はセグメントごとに分割数を変える実装（手前は細かく／遠方は粗く）にしたが、**継ぎ目に空が
線状に覗く不具合が出た**（ヘッドレス描画で発見）。原因：

- 手前セグメントの弧の奥端と、奥セグメントの弧の手前端は、同一平面・同一半径で接している。
- ここの分割数が違うと、**頂点数の違う内接多角形どうしは互いを包含しない**（powers-of-2に
  そろえても解決しない）。細かい側が粗い側の外へ僅かに張り出し、その隙間はどちらのポリゴンにも
  覆われず背景が抜ける。
- 「手前ほど細かい＝内接多角形が大きい＝手前が必ず覆う側になる」と考えたが誤り。手前セグメントの
  帯は接合面から**外側**へ、奥セグメントの帯は**内側**へ伸びるので、手前側が張り出すと逆に隙間が開く。

フレームに1つの分割数を共有すれば全ての継ぎ目が頂点単位で一致するため、この問題は原理的に起きない。
分割数が切り替わる瞬間の見た目の変化も許容量（0.8px）以下なので、走行中のポップも起きない
（z=8,000〜41,000を250刻みでスイープし、10→24へ1段ずつ推移することを確認）。

### 13-3. コスト実測（800x600・ヘッドレス、60フレームの最良値）

| 地点 | 旧（固定8） | 固定24 | 本実装（可変・上限24） |
|---|---|---|---|
| トンネルが遠い (z=10,000) | 6.84ms | 8.09ms | **7.12ms** (11分割) |
| 入口へ接近 (z=17,500) | 7.27ms | 8.56ms | **8.51ms** (22分割) |
| 入口直前 (z=20,050) | 8.41ms | 10.15ms | **10.21ms** (24分割) |
| 内部 (z=25,000) | 7.60ms | 9.00ms | **9.18ms** (24分割) |

最悪でも10.2msで、60fpsの予算16.6msに収まる。滑らかさが要る接近時は固定24と同等コストで、
トンネルが遠いフレームでは固定24より約1ms安い。滑らかさ/コストの調整は
`TUNNEL_ARC_TOLERANCE_PX` の1箇所で行う。

### 13-4. 不採用: 帯を1ポリゴンで描く案
セグメントの帯を「手前の弧＋奥の弧を逆順に連結した1枚のポリゴン」で描けば、描画呼び出しが
N分の1になって分割数を上げ放題になるはず、と考えて計測したが**1.1倍速くなるだけだった**
（N=24で57.3us→51.5us）。この描画はポリゴン数ではなく塗り面積で律速しており、呼び出し回数を
減らしても効かない。カーブで手前弧と奥弧の中心がずれると自己交差する危険もあるため不採用。

## 14. 山の描写を追加（2026-07-17）— 要件整理

現状はトンネルのアーチだけが平地に浮いて立っており、「なぜここにトンネルが要るのか」が読めない。
トンネルが埋まっている山（坑口の周りの地形）を追加する。

### 14-1. 決定事項（ユーザー確認済み）

- **形状の作り**: **入口面の平面シルエット**（1枚の多角形）。奥行きのある簡易3Dは採らない。
- **スケール**: **画面上部まで覆う大きな山**。接近時は空がほぼ隠れ、開口部だけが穴として残る。
  左右も画面外まで広げ、裾（山の左右の端）は接近時に見せない。
- **遠景の山並み**: 今回は入れない（スコープはトンネルの山1つ）。背景は現状の `bg1.png` のまま。

### 14-2. 実装方針（着手前のたたき台）

- **開口部は「穴」ではなく下辺の切り欠きになる**。アーチの根元は路面（＝山の裾と同じ平面）に
  接しているため、山の輪郭は *外周のシルエット → 右のアーチ根元 → アーチ輪郭を逆順 → 左のアーチ根元*
  と一筆でたどれ、**1枚の単純多角形**になる。pygameは穴あきポリゴンを描けないので、この性質が効く
  （12節の小口リングと同じ発想）。
- **描画位置**: 入口セグメント（既存の小口リングと同じ `(seg['p1']['z'] - STRIPE_LENGTH) < tunnel_start`
  の条件）の手前面上。既存の `tunnel_arc_pt()` に投影済みの原点 `(x1, y1)` とスケール `s1` を渡し、
  世界座標で定義した山の輪郭を同じスケールで投影する（道路・アーチと透視が自動で一致する）。
- **切り欠きの半径**: 坑内の弧（`TUNNEL_HALF_WIDTH`/`TUNNEL_HEIGHT`）ではなく、**小口リングの外枠**
  （`+TUNNEL_PORTAL_THICKNESS`）に合わせる。こうすると山とアーチの間に小口のコンクリート帯が残り、
  「山肌に開いた坑口」として読める。分割数は弧と同じ `tunnel_arc_n` を共有する
  （13-2の理由と同じ。ここだけ変えると継ぎ目に隙間が出る）。
- **描画順**: 坑内アーチ・天井ライトの後、**小口リングの前**。山が坑内を塞ぎ、その上からリングが
  開口部を縁取る。入口より手前のセグメント（道路・縁石）はBack-to-Frontループの後段で描かれるため
  山に覆われない。
- **輪郭の生成**: 決定論的な擬似乱数（固定シード）で稜線の点列を初期化時に1回だけ作ってキャッシュし、
  毎フレーム同じ形を描く。世界座標で持つので、遠方では小さく、接近すると画面を覆う形へ自然に育つ。
- **色**: まず単色（背景の地面と馴染む暗めの緑）で作る。フォグは坑内要素と違い、坑外なので
  小口面と同じ**通常の `fog_color`（空色）**へブレンドする。単色で平面的に見えるようなら、
  稜線ごとの明暗づけを後から検討する（アーチも単色から始めて足りている前例あり）。
- **コスト**: 追加は多角形1枚（頂点数＝稜線の点数＋`arc_n`）。13-3の実測から、この描画は塗り面積で
  律速するため、接近時に画面を覆う面積ぶんのコストは乗る。実装後に計測して16.6msの予算内を確認する。

### 14-3. 出口側の山を作らない理由

内部から出口を見たとき、開口部の向こうは開けた景色で正しい（出口面の山肌はカメラと反対を向く）。
通過後は振り返らないため、出口側の山は一度も画面に映らない。

### 14-4. 想定される破綻とリスク

- **平面シルエットなので視差が出ない**: 車が左右に動いても山の稜線は動かない。ただし山は入口面に
  貼り付いており、その面は道路がカーブしていてもz軸に垂直（既存のアーチと同じ扱い）なので、
  カーブ区間で破綻することはない。
- **裾が見えるのは遠方だけ**: 幅を「入口の手前2,500ユニットで画面幅を超える」ように取れば、
  接近時に裾が画面内に入らない。遠方で裾が見えるのは地形として自然なので許容する。

### 14-5. 実装ステップ

1. [x] 定数追加（山の高さ・幅・色・稜線の粗さ）と、稜線シルエットの生成＋キャッシュ
2. [x] `draw()` の入口セグメント処理に山の多角形を追加（坑内アーチの後・小口リングの前）
3. [x] ヘッドレス描画で確認（遠方／接近／入口直前／内部）＋コスト計測
4. [x] 実機プレイテスト

### 14-6. 実装メモ（2026-07-17）

- 追加定数（[track.py](../src/track.py) 冒頭）: `MOUNTAIN_HALF_WIDTH = 20000`（裾までの距離）、
  `MOUNTAIN_HEIGHT = 8000`（稜線の最大高さ）、`MOUNTAIN_COLOR = (46, 74, 44)`
  （※寸法は 14-7 で2倍に、色は 14-12 で (40, 92, 46) に変更済み。現行値はコードを参照）、
  `MOUNTAIN_SEED = 6`、`MOUNTAIN_DETAIL_LEVELS = 5`（2^5+1 = 33点）、`MOUNTAIN_ROUGHNESS = 0.35`。
- 稜線は `Track._build_mountain_ridge()` で `__init__` 時に1回だけ生成して `self._mountain_ridge`
  にキャッシュ（世界座標の (x, y) 列）。**釣鐘状の包絡線 × midpoint displacement のノイズ**という
  掛け算の形にしたのが要点：裾（包絡線=0）が必ず地面へ戻り、高さも包絡線を超えないので、
  「接近時にどれだけ画面を覆うか」を包絡線側の数字だけで見積もれる（ノイズを足す形だと両方が効いて
  裾が地面から浮く危険もある）。
- 寸法の根拠: 入口の手前2,500ユニット（scale = 300/2500 = 0.12）で路面の投影Yが
  `HORIZON_Y + CAMERA_HEIGHT*0.12 ≈ 480px`。画面上端を覆うには稜線が world 4,000 以上必要で、
  最も凹んだ場合（包絡線の65%）でも中心付近が 8,000*0.65 = 5,200 でこれを満たす。
  `MOUNTAIN_HALF_WIDTH = 20000` は同じ距離で左右の画面端（world ±3,333）まで高さを残すための幅。
- **コスト実測（800x600・ヘッドレス、40フレームの最良値。背景描画込み）**:

  | 地点 | 山なし | 山あり |
  |---|---|---|
  | 遠景 (z=10,000) | 8.62ms | 8.52ms |
  | 接近 (z=17,500) | 9.99ms | 9.94ms |
  | 入口直前 (z=19,900) | 11.65ms | 12.43ms |
  | 内部 (z=25,000) | 10.77ms | 10.46ms |

  増分は入口直前の約0.8msのみで、他は計測誤差の範囲。13-3で分かった「塗り面積で律速する」性質から
  多角形1枚の追加は実質ただで、覆う面積が最大になる入口直前だけコストが乗る。60fpsの予算16.6ms内。
- ヘッドレス描画4地点で確認済み：遠景では稜線がフォグで空色へ霞んで大気遠近になり、接近時は空が
  完全に隠れて開口部だけが穴として残る。入口直前・内部は山なしと同じ絵（山は背後に回るため正しい）。

### 14-7. 山を縦横2倍に拡大（2026-07-17）

`MOUNTAIN_HEIGHT` 8,000 → 16,000、`MOUNTAIN_HALF_WIDTH` 20,000 → 40,000。スタート地点から
トンネルが山として認識できるようになった。入口直前のコストは 12.4 → 13.3ms（覆う面積が増えたぶん）。
副作用として、接近時に稜線が画面外へ出て緑一色に見える距離帯が長くなった（z=10,000で既に頂上が
画面上端を超える）。走行中は一瞬なので現状維持。

### 14-8. 稜線のアンチエイリアス（2026-07-17）

稜線と空の境に1px単位の階段が見えていた。拡大して切り分けたところ、原因は**形の凹凸ではなく
ピクセルレベルのエイリアシング**だった（斜面はきれいな直線に階段が出るだけ。頂上付近は稜線が
ほぼ水平なため1px上がるまでの横の走りが長く、階段が最も目立つ）。

#### 14-8-1. 不採用: 分割数を増やす
逆効果。稜線は midpoint displacement で作っているため、段を増やすと**既存の曲線が細かく刻まれるのでは
なく、新しいランダムな凹凸が足される**（＝形はより粗くなる）。しかも追加される凹凸は1px前後なので、
スケールが変わるたびに稜線がチラつく。弧の `_arc_segments_for`（13節）は真円という収束先があるので
分割を増やせば滑らかになったが、**稜線には近似すべき真の曲線＝収束先がない**ので同じ理屈は通らない。

#### 14-8-2. 不採用: pygame.draw.aalines で稜線をなぞる
道路端の `EDGE_SMOOTHING`（[track.py](../src/track.py) 冒頭、現在は無効）と同じ手を試したが**効かない**。
`aaline` は「線」であってエッジのカバレッジAAではないため：
- 塗りつぶしの上から重ねても、塗りが真の稜線より上へはみ出して（内部で座標を丸める）AAの1pxの帯を
  食ってしまう。実測で `fill+aa` は `fill_only` と1ピクセルも違わなかった。
- 塗りを1px下げて帯を空けても、得られるのは**塗り色ほぼそのままの1px線**（例: 塗り(46,74,44)に対し
  中間色が(59,85,57)）。ほぼ水平な線では線自身のカバレッジがその行にほぼ全部乗るためで、
  階段の1pxの段差はそのまま残る。
- `pygame.gfxdraw.aapolygon` も内部は aaline の集合なので同じ。加えて座標が±32767を超えると破綻し、
  接近時の稜線（世界座標±40,000×スケール）は容易に超える。

#### 14-8-3. 採用: 稜線の帯を縦4倍で描いて縮小（`Track._blit_mountain_aa`）
`MOUNTAIN_AA_SUPERSAMPLE = 4`。稜線が画面に写っている**帯だけ**をSRCALPHAのSurfaceへ縦4倍で描き、
`smoothscale` で縮小して重ねる。縮小時のカバレッジ計算により、列ごとに連続的に変化する中間色が出る
（実測: 隣り合う列で (118,131,126) / (130,138,138) / (112,127,124) と変化＝階段が消える）。
- **ベースの塗りは稜線を1px下げて描く**。塗りが真の稜線より上へはみ出すと、その硬い縁はAAの帯
  （その部分は透明）では消せないため。下げたぶんはAAの帯が上から塗り直す。
- **帯は画面のx範囲に掛かる稜線からだけ求める**。接近時は稜線が画面の外（上）へ抜けて帯が空になり、
  AAは自動的にスキップされる。＝**山が画面を覆う一番重いフレームではコストゼロ**で、稜線が見えている
  軽いフレームでだけ払うという、必要な所にだけコストが乗る形になっている。
- コスト実測（800x600・ヘッドレス）: スタート地点 8.28→9.31ms（+1.03）、遠景 z=10,000 8.65→9.32ms
  （+0.67）、接近 z=17,500 +0.13ms（ほぼスキップ）、入口直前 z=19,900 は変化なし（スキップ）。
  z=0〜19,500を500刻みでスイープした最悪値は12.0ms（z=19,500、AAではなく塗り面積が要因）。予算16.6ms内。
- 残る「形の粗さ」（33点の折れ線としての角）はAAとは別問題。気になる場合は `MOUNTAIN_ROUGHNESS` を
  下げるか、段数を4に減らす調整になる。

### 14-9. 稜線を縦方向へにじませる（2026-07-17）

14-8のAAで階段は消えたが、にじむ範囲が1pxしかなく硬い。にじみを縦へ広げた。
`MOUNTAIN_AA_SUPERSAMPLE` を上げても**効かない**点に注意：あれは同じ1pxの中の階調数を増やすだけで、
にじむ「幅」は変わらない。幅を広げるにはぼかす処理そのものが要る。

- 新定数 `MOUNTAIN_RIDGE_FEATHER_PX = 4`（にじみの幅。1でほぼAAのみ、大きいほど空へ溶ける）。
- `_blit_mountain_aa` の縮小を「縦を feather 分の1へ潰してから戻す」に変更（11節のグローと同じ
  縮小→拡大ブラー）。**幅はWのまま触らない**ので横方向には滲まず、稜線の形は保たれる。潰す段が
  カバレッジ平均も兼ねるため、AA用の縮小と1回にまとめられる（＝14-8から処理は増えていない）。
- にじみが上下へ広がるぶん帯を `feather` だけ余分に取る（足りないとにじみが帯の縁で切れる）。
- **ベースの塗りの下げ幅も 1px → `feather + 1` px に変更**。塗りの硬い縁がにじみの中に残ると、
  そこで色が不連続に切り替わってにじみが台無しになるため。
- **重要な落とし穴: 帯の透明部分をRGBごと山の色で埋める**（`band.fill((*color, 0))`）。SRCALPHAの
  初期値 `(0,0,0,0)` のまま縮小すると、RGBとAが別々に平均されて縁のRGBが黒へ引っ張られ、にじみが
  黒く濁る。14-8の時点（1px）では気付きにくいが実際に出ていた（緑成分が正しい線形補間より9も暗かった）。
  RGBを一様にしておけばAだけが平均され、空との正しい線形補間になる（実測で確認済み）。
- **コストは feather の値によらずほぼ一定**（スタート地点で 1→9.34ms / 2→9.29 / 4→9.39 / 8→9.42ms）。
  にじみを広げるほど潰す先が小さくなるので、ブラーは実質ただ。標準地点も14-8から変化なし
  （遠景9.25ms・接近10.06ms・入口直前13.23ms）。
- **上限**: `feather = 8` にすると、潰した段の粗さがそのまま出てにじみ自体がブロック状に見え始める。
  4は滑らかで、これ以上広げたい場合は潰し方（多段化など）の見直しが要る。

### 14-10. にじみを縮小: 4px → 2px（2026-07-17）

14-9の4pxは「にじみすぎ」との指摘。実測で境界が約3px幅のなだらかな階調（206→173→135→120→110→山色）
に収まり、階段は出ないまま輪郭が締まる。`MOUNTAIN_RIDGE_FEATHER_PX = 4` → `2` の1行のみの変更
（コストはにじみ幅によらず一定という14-9の実測どおり、変化なし）。

### 14-11. にじみ幅を1px単位より細かく調整可能に（2026-07-17）

2pxからさらに狭めたい（1.5や1.75）との相談。`_blit_mountain_aa` は縮小サイズを `h // feather`
（整数除算）で求めていたため、小数のfeatherを渡しても丸めが粗く、狙いどおりの差が出なかった。

- **修正**: 帯の余白（整数pxで確保する必要がある）は `math.ceil(feather) + 1` で切り上げて別に持ち、
  実際のぼかしの強さ（縮小先のサイズ）は `round(h / feather)` で小数のfeatherをそのまま反映する
  ように分離した。これで余白は常に十分（にじみが帯の縁で切れない）まま、ぼかしの強さだけ小数で
  効くようになった。
- 実測（x=400の境界の色、空→山）: feather=1.0（旧硬い）は 206→157→108 と急に暗くなるのに対し、
  1.5 は 206→187→143→120→108、1.75 は 199→182→151→112→108、2.0（前段）は 206→173→135→120→110→107。
  狙いどおり1.0と2.0の間の段階になっている。コストは1.0〜2.0の全域で9.2〜9.4ms（誤差の範囲、
  14-9で確認済みの「にじみ幅によらず一定」を再確認）。
- **採用値: `MOUNTAIN_RIDGE_FEATHER_PX = 1.75`**。2pxよりわずかに締まった輪郭。

### 14-12. 山肌に森テクスチャを貼る（2026-07-17）

山肌は単色＋手続き的な濃淡ノイズだったが、のっぺりして見えるため **`asset/forest.png`（航空写真風）を
格子状に敷き詰める**方式へ置き換えた。あわせて `MOUNTAIN_COLOR` を (46, 74, 44) → **(40, 92, 46)** に
変更（テクスチャと重ねる前提で、地肌を背景の緑へ寄せた）。14-6 の定数一覧はこの値で読むこと。

- 追加定数（[track.py](../src/track.py) 冒頭）:
  `MOUNTAIN_FOREST_IMAGE = 'asset/forest.png'`、`MOUNTAIN_FOREST_TILE_WORLD_W = 4500.0`（タイル1枚の
  世界幅。s1倍してスクリーン上の寸法にする）、`MOUNTAIN_FOREST_ALPHA_MAX = 170`、
  `MOUNTAIN_FOREST_ALPHA_GAMMA = 0.7`。
- 描画は `Track._blit_mountain_forest()`。山の多角形と同じ輪郭でマスクするので、稜線やアーチの
  切り欠きをはみ出さない。

#### タイルの基準点は坑口の投影点にすること（bboxの角では駄目）
- タイルは**坑口中心の投影点 (anchor_x, anchor_y) を基準**に敷き詰める。
- **落とし穴**: マスクの左上（＝bboxの角）を基準にすると、接近してbboxが広がるたびに基準点自体が
  動くため、**模様が対角線状に流れて見える**。坑口中心はスクリーン上でほぼ動かないので、そこを
  基準にすれば模様は中心から均等に育つだけになる。

#### 濃さの決め方
- `overlay_alpha = ALPHA_MAX * (1 - fog_pct) ** ALPHA_GAMMA`。1次関数だと近距離で不透明になりすぎ、
  遠距離ではすぐ消えるため、`ALPHA_MAX`(170) で頭打ちにしつつ `ALPHA_GAMMA`(0.7 < 1) で遠距離の
  減衰を緩めている。地肌とブレンドさせる狙いもあり、至近でも完全な不透明にはしない。
- **テクスチャ専用のフォグ値を使う**: 通常のフォグには「水平線ブースト」（道路が消失点に近いかという
  *画面上の見た目*で決まる補正）が乗っており、これを使うと坑口からまだ遠い時点で 1.0 に張り付き、
  テクスチャが実際の距離より早く消える。そのため**ブースト前の値**（`mountain_forest_fog_pct`）を
  退避して渡している。
- タイル寸法の上限は画面幅。上限を設けないと、坑口への接近で scale が急増した際に `smoothscale` の
  対象が数千px四方まで膨らみ、そこだけ極端に重くなる（はみ出す分は敷き詰めループが別タイルで
  まかなうので、上限を付けても見た目は変わらない）。

#### 14-12-1. レビュー後の整理（2026-07-17）
- **不採用: `MOUNTAIN_FOREST_TILE_MIN_PX`（タイル寸法の下限）**。「遠方でタイル数が増えるのを抑える」
  という当初の目的が**そもそも成り立っていなかった**: 山のbboxとタイル寸法はどちらも s1 に比例して
  スケールするため、**タイル数は距離に依らず約95枚で不変**。下限は z≈24,000 より遠方でテクスチャ倍率を
  60pxに固定してパースの一貫性を壊すだけだった。削除の影響はその距離より遠方のみで、そこでは
  オーバーレイのアルファが既に100以下（実測の最大差分 13/255）。
- 格子の範囲計算を floor/ceil から**剰余**へ置き換え、anchorとの位相を保ったまま int で刻むようにした
  （z=500〜22,000 で画素単位の同一性を確認済み）。

## 15. トンネル区間で地平線ヘイズをフェードアウト（2026-07-17）

地平線付近を馴染ませるヘイズ類（最前面側のレイヤー）が、トンネル内では暗いアーチの上に
明るい横帯として浮いて見え、違和感の原因になっていた。対象は**4つ**あり、2系統に分かれる：

- **背景側**（[background.py](../src/background.py) `BackgroundManager`）: `_draw_gradient_band`（上段グラデ）、
  `_draw_lower_gradient`（下段グラデ）、`_draw_fog_gradient`（霧グラデ。Stage4/5のみ有効）。
- **最前面側**（[main.py](../main.py) 描画ループ内）: トラック描画の**後**に `HORIZON_Y` から80px下までを
  覆う霧オーバーレイ。`get_fog_color()` が背景画像から色をサンプリングするため**全ステージで常に有効**
  であり、トンネル内でもアーチ・坑口の山の上に描かれ続けていた（今回の違和感の主因）。

### 実装

- `BackgroundManager._compute_tunnel_haze_mult(player_z)` を新設し、ステージ設定の
  `tunnel_start_z`/`tunnel_length` から不透明度倍率（0〜1）を計算する。トンネル設定のない
  ステージでは常に1.0（既存ステージに影響なし）。
- 新定数（[background.py](../src/background.py) 冒頭）: `TUNNEL_HAZE_FADE_DISTANCE = 3000.0`（フェードに使う距離）、
  `TUNNEL_HAZE_HIDE_MARGIN = 1500.0`（境界の外側に取る完全非表示マージン）。
  入口では z = 15,500〜18,500 でフェードアウトし、手前1,500からトンネル内全域は完全非表示。
  出口側は対称（z = 42,500〜45,500 で復帰）。
  - マージンが必要な理由: 入口ちょうどで0になる設計だと、アーチが画面を覆う接近時（手前1,500前後）に
    まだ倍率が残り、暗部を横切る薄い帯が見えてしまう（ヘッドレス描画で確認して1,000→1,500に調整）。
- `BackgroundManager.draw()` に `player_z` 引数を追加（デフォルトNone＝従来動作）。倍率は
  `self._tunnel_haze_mult` に保持し、グラデ3種のalpha計算に乗算する。
- main.py側の霧オーバーレイも `bg_manager._tunnel_haze_mult` を参照して同じ倍率をalphaに乗算
  （倍率0のときはSurface生成ごとスキップ）。`bg_manager.draw()` の2箇所の呼び出しに `player_z=car.z` を追加。
- ヘッドレス描画で確認済み: トンネル内部・入口手前1,500で帯が消え、遠方（z=10,000）では
  倍率1.0強制と1ピクセルも差がない（誤発動なし）。Stage4/5など他ステージは常に倍率1.0。

### 15-1. 霧オーバーレイはステージ開始から即座に非表示へ変更（2026-07-17）

上記の入口側フェード窓（z=15,500〜18,500）でも、坑口の山は視認可能な距離（14-7の実測では
z=10,000で既に頂上が画面上端を超える）よりずっと手前から画面に写っており、フェード中の
薄い霧の帯が山のシルエットに重なって見える指摘があった。調べ直すと、Stage6は山が
**スタート地点（z=0）から常に視認できる**設計（14-7参照）のため、霧オーバーレイに関しては
「山が見える距離でフェード」ではなく「トンネル設定のあるステージでは開始から即座に非表示」
が実態に合う。

- `BackgroundManager._compute_overlay_haze_mult(player_z)` を新設し、main.py側の霧オーバーレイ
  専用の倍率にした。入口側はフェード窓を設けず、ステージ開始（z=0）〜トンネル通過
  〜出口+マージンまで一律0（即座に非表示）。出口側のみ`_compute_tunnel_haze_mult`と同じ
  フェード（`hide_end`から`TUNNEL_HAZE_FADE_DISTANCE`かけて1.0へ復帰）を流用する。
- 背景側の3種のグラデ帯（`_draw_gradient_band`/`_draw_lower_gradient`/`_draw_fog_gradient`）は
  従来どおり`_compute_tunnel_haze_mult`（入口手前からのフェード窓あり）を使う。山と重ならない
  高さにあるため、こちらは変更していない。
- `draw()`は`self._tunnel_haze_mult`（背景グラデ用）と`self._overlay_haze_mult`（霧オーバーレイ用）
  を両方計算して保持する。main.py側は`bg_manager._overlay_haze_mult`を参照するよう変更。
- ヘッドレス描画で確認済み: z=0時点で霧オーバーレイ倍率が既に0（山のシルエットに帯が重ならない）、
  出口通過後（z=46,000）は倍率1.0に復帰。

## 16. トンネルを3回登場・長さ3倍に変更（2026-07-17）

トンネル区間を1回・21,000ユニットから、**3回・63,000ユニット（各区間）**に変更した。

### 16-1. 決定事項（ユーザー確認済み）
- 長さ: 21,000 → 63,000（3倍）
- 登場回数: 1回 → 3回

### 16-2. 実装内容
- `STAGE_CONFIG[6]` の `tunnel_start_z`/`tunnel_length`（スカラー単発）を、
  `tunnels: [{'start_z':..., 'length':...}, ...]`（リスト）に置き換えた。
  配置は z = 20,000〜83,000 / 200,000〜263,000 / 400,000〜463,000（コース全長600,000ユニット、
  最終区間の後に137,000ユニットの余裕を残す）。
- [track.py](../src/track.py) 側の変更点:
  - `get_tunnel_at(z, stage_id)`: リストを走査し、いずれかの区間に含まれれば True。
  - `draw()`: `tunnel_start`/`tunnel_end`（スカラー）を `tunnel_ranges`（リスト）に変更。
    - `tunnel_arc_n`（弧の分割数）: 全区間を走査し、画面に写りうる区間があればその中で最大値を採用。
    - セグメントごとの `in_tunnel` 判定: そのセグメントが属する区間 `cur_tunnel = (start, end)` を
      特定してから判定する（複数区間があるため、単一のstart/endでは判定できない）。
    - 入口セグメント判定（小口リング・山の描画条件）: `cur_tunnel[0]`（そのセグメントが属する区間の

  - 弧の分割数・天井ライト・小口リング・山（14節）・路肩床（8-2節）は元々セグメント単位の描画のため、

- [background.py](../src/background.py) 側の変更点（15節のヘイズフェード）:
  - `_compute_tunnel_haze_mult` / `_compute_overlay_haze_mult` を、`tunnels` リストの各区間について
    従来と同じ計算式を適用し、**全区間の倍率のうち最小値**を採用する方式に変更（区間から離れていれば
    全区間が1.0を返すため、通常時の見た目は変化しない）。
  - `_compute_overlay_haze_mult` の入口側「隠し始める距離」は、従来の「ステージ開始z=0固定」から
    「各区間の`start_z - DRAW_DISTANCE`」に一般化した。1つ目の区間（z=20,000）はこの式でも
    `20,000 - 50,000 = -30,000`（負値＝実質z=0から）となり、従来の挙動と一致する。2つ目・3つ目の
    区間は、山が描画距離に入る手前（区間開始の50,000ユニット手前）から隠すようになる。
- ヘッドレス描画で3区間すべての通過（接近・入口・内部・出口・通過後）とヘイズ倍率の推移を確認済み。
  区間間（例: z=90,000〜149,000）では両倍率とも1.0に戻り、通常のStage1相当の見た目になる。

## 17. 坑内の明暗境界をぼかす（2026-07-17）

入口付近の明るさから奥の暗闇へ切り替わる勾配が急すぎたため、`TUNNEL_SHADOW_SOFTEN` を追加し、
坑内要素（アーチ・路面・路肩床・縁石）のフォグ合成に適用した。小口面と山は「外を向いた面＝空に
馴染ませる要素」なので対象外（従来のフォグのまま）。

### 17-1. 除算 → べき乗カーブへ変更・再チューニング（1.6 → 7.5）
- 初版は `fog_pct / SOFTEN` の除算だったが、これは**最奥の暗さそのものを 1/SOFTEN で頭打ちにして
  しまう**（奥が暗闇に到達しない）。
- `fog_pct ** (1.0 / SOFTEN)` のべき乗カーブに変更。手前から緩やかに暗くなり始めつつ、奥は 1.0
  ＝完全な暗闇へ到達する。この変更に伴い 1.6 → **7.5** へ再調整した。

### 17-2. 天井ライトだけ暗化を緩める（TUNNEL_LIGHT_SHADOW_RELIEF = 0.25）
- ライトは光源であって、壁と同じ速さで暗闇へ沈むと不自然なため、ライト本体・にじみにだけ
  `TUNNEL_SHADOW_SOFTEN` 適用後の暗化を緩める割合を掛ける。

## 18. 天井ライトのグローが壁・山を透ける不具合の修正（2026-07-17）

### 18-1. 症状と原因
- カーブ区間で、**奥のライトのグローが手前のアーチ壁や入口の山を透けて光り**、坑外から見たときに
  トンネルが半透明に見えていた。
- 原因: グローは専用のオフスクリーンSurfaceに描いてから1フレームに1回だけ加算合成しているため、
  **画面に描かれた何物にも遮蔽されない**（描画順による隠蔽が効かない）。

### 18-2. 実装内容
- グロー用Surfaceに対し、アーチの各面と山のシルエットを**実際の描画と同じ形状で黒く塗って消す**
  （黒＝グローなし）。これにより、その手前の形状を越えて実際に見えているライトのグローだけが残る。
- 消し込みパスが増えたぶんの塗りコストを相殺するため、`TUNNEL_LIGHT_GLOW_SURF_DOWNSCALE = 2` を
  追加してグロー用Surfaceを1/2解像度にした（にじみはどのみち縮小→拡大のブラーを通すので画質に
  影響しない）。

## 19. トンネル奥の端をキャップ（2026-07-17）

### 19-1. 症状と原因
- トンネル長（63,000）が `DRAW_DISTANCE`（50,000）より長いため、**本当の出口は描画されない**。
- 出口に見えていたものは、実際には筒が描画限界で切り落とされた断面であり、そこからアーチ越しに
  空・地平線・草地の背景が覗いていた。結果、ありえないほど近くに出口があるように見えていた。

### 19-2. 実装内容
- 最遠の可視断面を `TUNNEL_FOG_COLOR` で塗り潰す。その位置では `fog_pct` が 1.0 に達しており、壁も
  既に同じ色になっているため、キャップは継ぎ目なく馴染む。
- **本当の出口が描画距離に入ったらキャップをスキップ**する（出口の外光が見えるようにするため）。

## 20. 坑口から差し込む外光（2026-07-17）

### 20-1. 症状
- 坑口（入口・出口とも）のすぐ内側のセグメントが、坑内の最奥と同じ一様な暗さになっており、外光が
  届くはずの開口部直近が不自然だった。

### 20-2. 実装内容
- セグメントの**坑口からの奥行きだけ**で決まる `daylight` 係数を追加（`TUNNEL_DAYLIGHT_REACH =
  4500.0` にかけて線形に0へ減衰）。
- **重要**: これは `TUNNEL_SHADOW_SOFTEN`（＝カメラからの距離で決まる）とは独立している。走って
  近づいても明るさが動かないようにするための分離であり、両者を混ぜてはいけない。
- 面ごとに当て方が異なる:
  - **路面・床**: 「トンネル外と同じフォグ計算をした色」へブレンドする。坑口（奥行き0）では外の
    色に完全一致し、外の路面とシームレスに繋がる。
  - **アーチ**: 外に対応物がなくブレンド先がないため、暗化を緩めて `TUNNEL_ARCH_COLOR` へ近づける
    だけ。床だけ明るいと不自然なので `TUNNEL_DAYLIGHT_ARCH_RELIEF = 0.45` で弱めに追従させる。
  - **天井ライト**: 光源であって外光を受ける面ではないため適用しない。

## 21. アーチ・床の色を暖色化（2026-07-17）

天井ライトの環境光の反射・疑似グローを想定し、コンクリートのベース色をわずかに暖色へ寄せた。

### 21-1. 不採用: B を引いて色相を寄せる方向
- 最初の試行は**Bチャンネルを削って**色相を寄せたが、**赤黒く暗く見える**。青を抜くと知覚輝度が
  下がるためで、「暖色」ではなく「汚れ」に見えてしまう。
- 採用した方向: **R/G を加算し、B はそのまま**。明るさを保ったまま暖色を足せる。強さは最初の試行の
  約半分に調整した。
- 結果: `TUNNEL_ARCH_COLOR = (60, 55, 57)` / `TUNNEL_FLOOR_COLOR = (77, 72, 74)`
  （ニュートラルグレー (55, 53, 57) からの加算）。

## 22. 横方向の制限と壁擦りの火花（2026-07-17）

車体が端から端まで移動できるため、**トンネルから横に抜け出せて**背景が見えていた。

### 22-1. 決定事項（ユーザー確認済み）
- 厳密な制限でなくてよいが、抜け出しにくくする（どのみち抜け出すほど横にずれると路肩によるスピード
  ダウンのペナルティがあるため、通常のプレイでは試されない）。
- 壁の位置は中心から **1200**（`TUNNEL_WALL_LIMIT`）。
- 壁に当たっている間は、高速コーナリングと同じ火花を出す。

### 22-2. 壁の位置をアーチの実寸で決めてはいけない
- アーチの根元は `TUNNEL_HALF_WIDTH` = 2371.6 だが、**カメラのxが車のxそのもので、かつ画面の視野が
  アーチ幅より広い**ため、車体が根元へ届くはるか手前からカメラが坑外を映す。
- ヘッドレス描画で、カメラxに対して画面に占める空の割合を実測した値（坑内の最奥、外光の届かない
  位置。空の色を判別可能な色で塗ってからマスクで画素数を数えた）:

  | カメラx | 空の割合<br>800x600（実解像度） | 空の割合<br>1280x720（参考） | 備考 |
  |---|---|---|---|
  | 800 | 0.0% | 0.0% | |
  | 1000 | 0.0% | 0.3% | |
  | 1200 | **0.1%** | 4.0% | **採用** |
  | 1400 | 5.5% | 13.2% | |
  | 1600 | 20.8% | 28.2% | |
  | 1966 | 57.5% | 52.1% | 根元から車体幅ぶん内側。**車体幅ぶんの余裕では全く足りない** |
  | 2249 | 63.6% | 57.2% | 根元から車体の半幅ぶん内側 |

- **この値は解像度・アスペクト比に依存する**。ゲームは800x600固定（[main.py](../main.py) の
  `SCREEN_WIDTH`/`SCREEN_HEIGHT`、`set_mode` にスケーリングなし）なので左列が実際の値。
  **解像度を16:9へ変更する場合は 1200 でも約4%の空が見えるようになるため、この表を測り直すこと。**
- **1000 は不採用**: 実解像度ではどのみち 1200 でほぼ空が見えない（0.1%）うえ、1000 はオフロード
  判定の境界（約1025）より内側になり、トンネル内だけ路肩のスピードダウンが起きなくなる
  （＝抑止力が消える）。1200 は空を消しつつ路肩ペナルティも残せる位置。

### 22-3. 実装内容（[car.py](../src/car.py) / [track.py](../src/track.py) / [main.py](../main.py)）
- **座標のクランプではなく毎フレームの押し戻し**にしている。坑外は横方向が無制限なので、大きく外へ
  逃げてから突入した場合、クランプだと入口で瞬間移動になるため。
  - 押し戻し量 = `max(TUNNEL_WALL_PUSHBACK, めり込み量 * TUNNEL_WALL_PUSHBACK_RATIO)`
    = `max(20.0, overshoot * 0.2)`。
  - 最小20は最大ステア量（`STEER_SENSITIVITY_LOW` = 12）＋ドリフトを上回るので、壁際では事実上の
    固い壁になる。
  - **比例項が必要な理由**: 固定20だけだとステアの12と拮抗して差し引き8/フレームしか戻らず、入口
    手前から全開で横へ逃げてから突入すると（x=14,605まで到達可能）**壁に埋まったまま約22秒**走れて
    しまう。比例項を足して約0.38秒に短縮した。
- `Car.wall_contact`（0=非接触 / -1=左 / +1=右）を毎フレーム更新し、火花の発生に使う。単に端にいる
  かではなく**壁を押しているか**で判定する（擦っていなければ火花は出ないため）。
- **火花の閾値は `WALL_SPARK_MIN_SPEED` = 10.0**（路肩の砂埃と同じ）。
  - **落とし穴**: 壁（1200）はオフロード境界（約1025）より外にあるため、壁に触れている間は必ず
    オフロード扱いになり、速度が `OFFROAD_MAX_SPEED`（32.4 ≈ 67km/h）に張り付く。つまり「中速域で
    壁を擦る」状態はほぼ存在しない。**32.4 より上げると火花はほぼ出なくなる**（中速域=48.5 にすると
    高速で突っ込んだ直後の減速中0.4秒しか出ない）。
- 火花の見た目・位置は高速コーナリングのものと同一（`bottomright/left ±15`、`bottom - 20`、接触側で
  flip）。ただし接触は連続するため、発生頻度は既存の 0.03 より高い 0.25 にしている。

### 22-4. 検証
- ヘッドレスで3区間すべて・左右両方向について、アクセル全開のまま壁へ押し付け続けても
  `|x|` が 1200 を超えないことを確認済み。
- 同条件で路肩のスピードダウンが従来どおり働くことも確認（約34秒間オフロード判定、速度は
  162 → 26.8 まで低下）。トンネル外では `wall_contact` が一度も立たないことも確認済み。
