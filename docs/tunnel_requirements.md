# トンネル機能 要件整理（確定・実装着手前）

作業ブランチ: `feature/tunnel-gimmick`（mainには影響なし、実装前の要件整理段階）

## 1. 決定事項（ユーザー確認済み）

- **機能範囲**: 単発ギミック**背景**（見た目のみ）。物理挙動（グリップ・ブレーキ等）には影響させない。
  - Stage4（砂）・Stage5（ウェット）のような `car.py` 側の `if stage_id == N` 分岐は追加しない。
- **配置方式**: **新規専用ステージ（Stage6）**として追加する。中身は **Stage1の設定をコピー**したものをベースとし、コースの**一部区間**にトンネル演出を挿入する（Stage6全体をトンネルにはしない）。
  - `STAGE_CONFIG[6]` は `STAGE_CONFIG[1]`（環境色・カーブ特性・縁石あり等）を複製した内容にする。
  - ※ `create_road()` は `stage_id` を乱数シードに使っている（[track.py:170](../src/track.py:170) `rnd = random.Random(s_id)`）ため、設定値をStage1と同じにしても、Stage6は**Stage1と同じ配色・同じ難易度傾向だが、実際のカーブ順序はStage1とは異なる別コース**になる。「見た目・難易度の系統がStage1と同じ」という意味でのコピーであり、コースレイアウトの完全複製ではない点に注意（もし完全に同一レイアウトが必要な場合はシード固定の追加対応が必要）。

## 2. 現状把握（コード調査結果）

- `asset/bg6.png` は既に存在するが、中身は **HORIZON_Y=300 のキャリブレーション用グリッド画像**であり、トンネル用アセットではない。ただし3-3の方針（手続き的な図形描画で天井・側壁を表現）を採用する場合、これは無視してよい（Stage6の背景画像自体はStage1の `bg1.png`/`bg1v.png` をそのまま参照する想定）。
- ステージ追加の影響箇所（データ駆動な設計のため、想定より変更点は少ない）：
  - [track.py](../src/track.py) `STAGE_CONFIG` に `6:` エントリを追加（コース特性・環境色）。
  - [main.py:260](../main.py:260) `if stage_id > 5:` → `> 6` に変更（GAME_CLEAR判定）。
  - [main.py:602](../main.py:602) `if render_stage_id > 5: render_stage_id = 5` → `6` に変更（リプレイ等での描画クランプ）。
  - `stage_times` / ランキング集計は辞書ベースで stage_id 非依存のため追加変更不要。
  - UI側（[ui.py](../src/ui.py)）もステージ数をハードコードしていないため追加変更不要（「TOP 5 RECORDS」はランキング順位表示でステージ数とは無関係）。
- トンネルの「壁・天井」表現は現状のレンダリングパイプラインに **該当するレイヤーが存在しない**：
  - [background.py](../src/background.py) は空（`BackgroundLayer`）と地面（`GroundLayer`）の2層のみ。天井や側壁を隠す/描画する仕組みはゼロから追加する必要がある。
  - 道路描画（[track.py:305](../src/track.py:305) `draw()`）はセグメントのポリゴン塗りのみで、縁石の外側に構造物を描くフックはない。

## 3. 未決事項（実装着手前に方向性を決めたい点）

### 3-1. トンネル区間の位置・長さ・検出方法（決定：固定位置・固定長）
- 毎回同じ地点・同じ長さでトンネルが発生する（ランダムではない）。
- **発生位置**: 序盤（スタート直後）。スタートライン直後の平坦区間（50セグメント＝15,000ユニット、[track.py:173](../src/track.py:173)）を抜けた後、`z = 20,000` からトンネル開始とする案（コース全長600,000ユニットの3.3%地点）。
- **長さ**: 標準＝約70セグメント（約21,000ユニット）。`z = 20,000〜41,000` がトンネル区間。
- 実装上、`STAGE_CONFIG[6]` に `tunnel_start_z: 20000` / `tunnel_length: 21000` を追加し、`Track` に `get_tunnel_at(z)` のような判定関数を新設する（Stage4の縁石判定 [track.py:263](../src/track.py:263) `get_curb_at` と同様のパターン）。

### 3-2. 視覚表現の内容（決定：最小構成＝天井・側壁のみ）
- 含める: 天井・側壁の描画（空を隠し、暗いトンネル内部を表現）。
- 今回のスコープ外（将来の拡張候補として保留）: 等間隔の照明、入口・出口の明暗フェード、霧色の特別な切り替え。
- **実装方針（たたき台）**: 天井・側壁は新規画像を使わず、`track.py` の `draw()` ループ内で道路ポリゴンと同じ透視スケール（`s1`/`s2`）を使い、トンネル区間中のセグメントにのみ暗色の台形ポリゴン（天井1枚＋左右の壁2枚）を重ね描きする。これにより新規アセットが不要になり、コード追加も `track.py` の描画ループ内で完結する見込み。

### 3-3. アセット
- 上記たたき台（手続き的な図形描画）を採用する場合、**新規のトンネル専用背景画像は不要**。Stage6のベース背景（`bg1.png`/`bg1v.png` を流用、またはコピー）はそのままで良い。
- もし将来的に画像ベースの壁面テクスチャ等を追加したくなった場合は別途検討。

### 3-4. コード規模
- 最小構成（天井・側壁の図形描画のみ）であれば `track.py` への追加は数十行程度と見込まれ、500行制限に抵触する可能性は低い。実装時に改めて行数を確認する。

## 5. ステップ3の実装メモ（2026-07-17）

- 追加した定数（[track.py](../src/track.py) 冒頭）: `TUNNEL_HEIGHT`（天井高さ3000 world units）、`TUNNEL_HALF_WIDTH`（壁位置＝道路端の15%外側）、`TUNNEL_WALL_COLOR` / `TUNNEL_CEILING_COLOR` / `TUNNEL_FOG_COLOR`。
- 描画は `draw()` の Back-to-Front ループ内、縁石描画の後に挿入。セグメントごとに左右の壁（縦面）＋天井（見上げ面）の3ポリゴンを、道路と同じ透視スケール（`s1`/`s2`）で描く。追加コードは全体で50行弱（500行制限に十分収まる）。
- **要件整理時から1点追加した対応**: トンネル区間中の路面・縁石のフォグ到達色を `TUNNEL_FOG_COLOR`（暗色）に切り替えた。ヘッドレス描画で確認したところ、通常のフォグ（空色）のままだと暗いトンネル内で遠方の路面だけ水色に発光して見える不整合があったため。スコープ外にした「霧演出（入口・出口フェード等）」とは別の、描画整合性のための最小限の変更。
- **既知の挙動（プレイテストで要確認）**: 出口の手前約1,500ユニットでは、残りの壁・天井が視野角の外に投影されるため画面から消える（透視投影として正しい挙動で、実際の運転でも出口直前は開口部しか見えないのと同じ）。走行速度では一瞬なので問題ないと想定しているが、気になる場合は将来の照明・フェード演出（3-2の保留項目）とあわせて調整する。
- トンネル内の道路脇に地面（草）の緑が細く見えるのは既知。路肩として自然に読めるため現状維持（気になる場合は壁の根元と道路端の間に暗色の床面ポリゴンを足す拡張が可能）。
- ヘッドレス描画テスト（接近/入口/内部/出口手前/通過後の5地点）で目視確認済み。テストスクリプトはスクラッチパッド（`render_tunnel_test.py`）にあり、リポジトリには含めていない。
