# トンネル機能 要件整理（確定・実装着手前）

作業ブランチ: `feature/tunnel-gimmick`（mainには影響なし、実装前の要件整理段階）

## 1. 決定事項（ユーザー確認済み）

- **機能範囲**: 単発ギミック**背景**（見た目のみ）。物理挙動（グリップ・ブレーキ等）には影響させない。
  - Stage4（砂）・Stage5（ウェット）のような `car.py` 側の `if stage_id == N` 分岐は追加しない。
- **配置方式**: **新規専用ステージ（Stage6）**として追加する。中身は **Stage1の設定をコピー**したものをベースとし、コースの**一部区間**にトンネル演出を挿入する（Stage6全体をトンネルにはしない）。
  - `STAGE_CONFIG[6]` は `STAGE_CONFIG[1]`（環境色・カーブ特性・縁石あり等）を複製した内容にする。
  - ※ `create_road()` は `stage_id` を乱数シードに使っている（[track.py:170](../src/track.py:170) `rnd = random.Random(s_id)`）ため、設定値をStage1と同じにしても、Stage6は**Stage1と同じ配色・同じ難易度傾向だが、実際のカーブ順序はStage1とは異なる別コース**になる。「見た目・難易度の系統がStage1と同じ」という意味でのコピーであり、コースレイアウトの完全複製ではない点に注意（もし完全に同一レイアウトが必要な場合はシード固定の追加対応が必要）。

## 6. 断面形状を半楕円アーチに変更（2026-07-17）

直方体（垂直な壁2枚＋平らな天井1枚）から、円柱を横に半分に割ったような半楕円の弧に変更。
- 従来の `TUNNEL_HEIGHT`（アーチ頂点の高さ）・`TUNNEL_HALF_WIDTH`（路面との接点までの幅）はそのまま弧の縦横の半径として流用。幅と高さを別々に持つため厳密な真円ではなく半楕円（円柱っぽく見せる目的で幅・高さは独立調整可能な方を採用）。
- 弧は `TUNNEL_ARC_SEGMENTS = 8` 分割の台形ポリゴンで近似（`theta = 0〜π` を8分割し、`x = HALF_WIDTH*cos(theta)`, `y = HEIGHT*sin(theta)` で断面上の点を計算）。
- 面ごとの陰影（明暗の作り分け）は行わず、`TUNNEL_ARCH_COLOR` 単色（フォグのみ通常通り距離でブレンド）。旧 `TUNNEL_WALL_COLOR`/`TUNNEL_CEILING_COLOR` の2色使い分けは廃止し、`TUNNEL_ARCH_COLOR` に一本化。
- ヘッドレス描画で再確認済み。アーチの輪郭は8分割のファセットがわずかに見えるが、通常のプレイ距離では滑らかな半円状に見える。

## 7. 天井ライトの追加（2026-07-17）

### 7-1. 決定事項（要件整理）
- **見た目**: 単色の平面（天井上の明るい図形のみ）。グロー・にじみ演出は入れない。
- **配置間隔**: 密（短い間隔で連続的に見える）。
- **路面への影響**: 入れない（天井の装飾のみ、路面色・車体は変化しない）。

### 7-2. 実装内容
- 追加定数（[track.py](../src/track.py) 冒頭）: `TUNNEL_LIGHT_COLOR`（暖色系）、`TUNNEL_LIGHT_ARC_RANGE = (3, 5)`（アーチ頂点付近2ファセットのみ点灯対象）、`TUNNEL_LIGHT_SPACING = 4`（4セグメント周期）、`TUNNEL_LIGHT_ON_LENGTH = 2`（周期のうち2セグメント点灯＝密な配置）。
- 半楕円アーチのファセット描画ループ内で、`seg['index'] % TUNNEL_LIGHT_SPACING < TUNNEL_LIGHT_ON_LENGTH` の場合のみ、頂点付近のファセット色をアーチ色からライト色に切り替える。新規ジオメトリは追加せず、既存のアーチ面を塗り分けるだけなので実装コストが小さい。
- ライト色も通常のフォグ処理（`fog_pct`によるブレンド）を通すため、奥のライトほど暗闇に沈んで見える（近くのライトほど明るい）。
- ヘッドレス描画で確認：トンネル内部からは、頂点付近に山形のライトが等間隔に連なって奥へ消えていく見た目になっている。

### 7-3. 調整：左右2灯に分離・縮小（2026-07-17）
- 元の実装は弧のファセット（`TUNNEL_ARC_SEGMENTS`分割）をそのまま点灯色に塗り替える方式だったため、頂点を挟んだ左右のファセットが隣接し「1本の帯」に見えていた。
- 弧のファセット分割とは独立した角度でライト用の四角形を重ね描きする方式に変更（`tunnel_arc_pt(theta, ...)` ヘルパーを追加）。これにより弧の滑らかさ（8分割）を変えずにライトのサイズ・位置を自由に調整できる。
- 新定数: `TUNNEL_LIGHT_CENTER_OFFSET`（頂点からライト中心までの角度、20°）、`TUNNEL_LIGHT_HALF_ANGLE`（ライト自体の半幅、7°＝全体14°）。左右対称に2灯を配置し、中心に約26°分のギャップができる。
- サイズは旧実装（1本45°）に比べて1/3程度に縮小。
- ヘッドレス描画で再確認済み：左右にはっきり分かれた小さめのライトが奥へ連なる見た目になっている。

- `TUNNEL_HEIGHT` を 3000 → 2400（20%低く）に変更。`TUNNEL_HALF_WIDTH`（約1948）は据え置き。縦横比（高さ÷半幅）は 1.54 → 1.23 になり、縦長のまま横長寄りへ近づいた形。

## 11. グローを縮小→拡大ブラー方式へ作り直し（2026-07-17）

10-②の方式（本体より広い角度の半透明ポリゴンを2段階重ねる）では、輪郭のはっきりした台形が重なるだけで
「オレンジのライトを色違いの黄色いパネルが囲んでいる」ようにしか見えず、発光感・にじみ感が出なかった。
角度を変えた多層ポリゴンには連続的な減衰がないことが原因なので、ブラーを実際にかける方式へ変更した。

- **ライト本体**: 変更なし。従来どおり台形ポリゴンを直接スクリーンへ描く（輪郭はシャープ）。
- **発光用Surface**: `self._tunnel_glow_surf` を `SRCALPHA` から**黒で塗り潰した不透明Surface**へ変更。加算合成では黒＝発光なしとして扱われるため、縮小時にライトの色が周囲の黒と混ざって減衰が生まれる。ここへは**ライト本体と同じ形のポリゴンだけ**を描く（広げたポリゴンは不要）。
  - 描画色は `interpolate_color(TUNNEL_LIGHT_COLOR, (0,0,0), fog_pct)`。奥ほど黒へ落ちるので、`glow_fade` のような明示的な減衰計算は不要になり削除した。
- **疑似ブラー** (`Track._blit_tunnel_glow`): 発光用Surfaceを `pygame.transform.smoothscale` で縮小→拡大する。拡大時の補間により、多層ポリゴンでは作れない連続的な減衰＝にじみが得られる。
  - 新定数 `TUNNEL_LIGHT_GLOW_LEVELS` = `((4, 1.0), (12, 0.7))`（`(縮小率, 強さ)` を縮小率の小さい順に並べる）。縮小率が大きいレイヤーほど広く弱いにじみになる。各レイヤーは**前段の縮小結果からさらに縮小**するので、一気に縮小するより滑らかで安い。
  - 強さの調整は `fill((w,w,w), BLEND_MULT)` による乗算で行う。加算合成ではアルファ値が無視されるため、`set_alpha` は使えない。
  - 各レイヤーを `smoothscale` で全画面へ拡大し、`BLEND_ADD` でスクリーンへ重ねる。強さ1.0のレイヤーが重なる中心は白飛びし、光源の芯として見える。
  - 縮小/拡大の中間Surfaceは `self._glow_scratch` にサイズ別でキャッシュし、毎フレームの確保を避ける。
- ヘッドレス描画（トンネル内部3地点）で確認済み：滲みが連続的に減衰し、近距離のライトは芯が白飛びして発光しているように見える。パネル状の見えは解消。
- 実測コストは約0.6ms/frame（トンネル内 7.4ms / トンネル外 6.8ms、800x600・ヘッドレス）。60fpsの予算16.6msに対して十分小さい。

## 12. 入口に厚み（小口面）を追加（2026-07-17）

アーチが薄い板に見えてトンネルとして不自然だったため、入口坑口に厚みの見える面を追加した。

- **構成**: 入口セグメント（`tunnel_start` をまたぐ最初のセグメント）の手前面上に、外枠アーチ（半径 `+TUNNEL_PORTAL_THICKNESS`）と内枠アーチ（＝坑内の弧と同じ半径）の二重ポリゴンを取り、間を `TUNNEL_ARC_SEGMENTS` 分割の四角形で埋める。既存の `tunnel_arc_pt()` に半径引数（`hw`/`h`）を追加して流用した。
- 新定数: `TUNNEL_PORTAL_THICKNESS = 170.0`（`ROAD_WORLD_WIDTH`=約10m換算で約50cm相当）、`TUNNEL_PORTAL_COLOR = (120, 118, 122)`（外光に照らされたコンクリート想定。坑内のアーチ色より明るくして厚みが読めるようにした）。
- 小口面は坑外（外光側）にあるため、フォグ到達色は他のトンネル要素と違い `TUNNEL_FOG_COLOR`（暗闇）ではなく**通常の `fog_color`（空色）**へブレンドする。周囲の昼光と馴染む。
- 描画順は坑内アーチ・天井ライトの後（同じセグメントのループ内）。半径が坑内の弧より外側なのでZファイティングは起きない。
- **奥行き方向へは延ばしていない**: 当初は「外アーチ→少し奥へオフセットした内アーチ→側面を四角形で接続」という押し出し方式で実装したが、内枠から奥の空間は坑内の弧そのものが既に埋めており冗長だった。加えて入口手前1,000ユニット以内ではその側面を極端な斜め（ほぼ真横）から見ることになり、画面の周辺全体が明るい小口色で覆われて「灰色の壁に突っ込む」見えになった。入口面上の平らなリング1枚に変更してこの破綻を解消（ヘッドレス描画で入口手前600ユニットの絵が変更前＝リングなしと完全一致することを確認）。
- ヘッドレス描画5地点（手前6,000／2,500／600／100ユニット・内部）で確認済み：接近時に開口部を縁取るコンクリートの帯として見え、遠方では細い明るい輪郭、至近距離・内部では破綻なし。

## 13. 断面を滑らかに（弧の分割数を距離から決める）（2026-07-17）

固定8分割では接近時にアーチの輪郭が多角形に見えていた（6節参照）。一律に分割数を上げると
トンネルが遠いフレームまで重くなるため、**フレームごとに必要な分割数を計算する**方式にした。

### 13-1. 実装
- `TUNNEL_ARC_SEGMENTS = 8`（固定値）を廃止し、`Track._arc_segments_for(radius_px)` を新設。
  N分割の内接多角形は弦の中央で真円より `sagitta = R*(1-cos(pi/2N))` だけ凹む。これが多角形に
  見える正体なので、凹みが許容値以下になる最小のNを小角近似 `sagitta ≈ R*(pi/2N)^2/2` から逆算する。
- 新定数: `TUNNEL_ARC_TOLERANCE_PX = 0.8`（凹みの許容量。小さいほど滑らかで重い）、
  `TUNNEL_ARC_SEGMENTS_MIN = 4`、`TUNNEL_ARC_SEGMENTS_MAX = 24`。
- 基準半径は**最寄りのトンネル断面**の投影後の半径（＝画面上で最も大きく写り、多角形が最も目立つ位置）。
  結果の `tunnel_arc_n` を描画ループの前で1回だけ求め、全トンネルセグメントで共有する。
- `MAX = 24` の根拠: 接近時（入口手前2,500）に必要なのは22分割なので画質には影響せず、
  弧が画面外まで広がる至近距離のコストだけを抑える。

### 13-2. 重要: 分割数はセグメントごとに変えてはいけない
最初はセグメントごとに分割数を変える実装（手前は細かく／遠方は粗く）にしたが、**継ぎ目に空が
線状に覗く不具合が出た**（ヘッドレス描画で発見）。原因：

- 手前セグメントの弧の奥端と、奥セグメントの弧の手前端は、同一平面・同一半径で接している。
- ここの分割数が違うと、**頂点数の違う内接多角形どうしは互いを包含しない**（powers-of-2に
  そろえても解決しない）。細かい側が粗い側の外へ僅かに張り出し、その隙間はどちらのポリゴンにも
  覆われず背景が抜ける。
- 「手前ほど細かい＝内接多角形が大きい＝手前が必ず覆う側になる」と考えたが誤り。手前セグメントの
  帯は接合面から**外側**へ、奥セグメントの帯は**内側**へ伸びるので、手前側が張り出すと逆に隙間が開く。

フレームに1つの分割数を共有すれば全ての継ぎ目が頂点単位で一致するため、この問題は原理的に起きない。
分割数が切り替わる瞬間の見た目の変化も許容量（0.8px）以下なので、走行中のポップも起きない
（z=8,000〜41,000を250刻みでスイープし、10→24へ1段ずつ推移することを確認）。

### 13-3. コスト実測（800x600・ヘッドレス、60フレームの最良値）

| 地点 | 旧（固定8） | 固定24 | 本実装（可変・上限24） |
|---|---|---|---|
| トンネルが遠い (z=10,000) | 6.84ms | 8.09ms | **7.12ms** (11分割) |
| 入口へ接近 (z=17,500) | 7.27ms | 8.56ms | **8.51ms** (22分割) |
| 入口直前 (z=20,050) | 8.41ms | 10.15ms | **10.21ms** (24分割) |
| 内部 (z=25,000) | 7.60ms | 9.00ms | **9.18ms** (24分割) |

最悪でも10.2msで、60fpsの予算16.6msに収まる。滑らかさが要る接近時は固定24と同等コストで、
トンネルが遠いフレームでは固定24より約1ms安い。滑らかさ/コストの調整は
`TUNNEL_ARC_TOLERANCE_PX` の1箇所で行う。

### 13-4. 不採用: 帯を1ポリゴンで描く案
セグメントの帯を「手前の弧＋奥の弧を逆順に連結した1枚のポリゴン」で描けば、描画呼び出しが
N分の1になって分割数を上げ放題になるはず、と考えて計測したが**1.1倍速くなるだけだった**
（N=24で57.3us→51.5us）。この描画はポリゴン数ではなく塗り面積で律速しており、呼び出し回数を
減らしても効かない。カーブで手前弧と奥弧の中心がずれると自己交差する危険もあるため不採用。
