### 注意事項

#### 1. 新規実装について
- バージョン管理はgitのコミット履歴で行う（フォルダ単位の管理は廃止済み）。
- 新規実装は `main.py` / `src/` を直接編集して進めてよい。
- **動作確認が取れるまではコミットしない**。ユーザーが動作を確認し、問題なければ明示的な指示があった場合のみコミットする（これまでの「Stable版への昇格はユーザー判断のみ」というルールをコミット単位に置き換えたもの）。
- ROADMAP の各課題 [ ] は一つずつ順番に処理し、各ステップごとにユーザーの指示を確認してから次に進むこと。
- コミット操作・GitHubへのpushは AI から自動で行わない。必ずユーザー判断を待つ。

#### 2. コミット時に必ず実施する作業（バージョン管理）
ユーザーがコミットを指示したら、以下を確認・更新すること：

1. **コードコメント**
   - `main.py` 1行目に「ver名」
   - 2行目に「現在の日付（YYYY-MM-DD）」

2. **ゲーム内表示タイトル**
   - タイトル画面またはバージョン表示欄に ver名を反映する。

3. **ROADMAP**
   - 更新点を1行で追記する。
   - 形式例：「YYYY-MM-DD　ver名　更新内容」
   - ROADMAPに重複箇所があればその部分を削除

4. **gitタグ（任意）**
   - 大きな区切り（旧Stable版相当）には `git tag v25_stable` のようにタグを付けておくと、フォルダ名管理をしなくてもその時点へ戻れる。

※ `asset/`（画像・音声）はgit管理対象外。アセットを追加・変更した場合はディスク上に配置するだけでよく、コミットには含まれない。

#### 3. フェーズ開始時の確認
- 疑問点や不明点があれば、開発者に直接質問すること。

#### 4. コード規模の制限
- 新規実装コードが **500行を超える**、または超える可能性が高い場合、AI は作業前にユーザーへ分割検討を提案すること。

#### 5. ユーザーへの報告言語
- すべて日本語で報告する（変数やメソッド名などは英語で構わない）

#### 6. デバッグコード規約
デバッグ用コードは以下の規約に従い、後で簡単に削除できるようにする：

- **変数名**: `debug_***` を使用（例: `debug_speed`, `debug_gravity_accel`）
- **ログ出力**: `DBG:` を先頭につける（例: `print("DBG: speed=", debug_speed)`）
- **コメント**: `# DEBUG YYYY-MM-DD: 説明` を必ず付ける
- **複数行の場合**:
  ```python
  # <DEBUG YYYY-MM-DD>
  debug_value = some_calculation()
  print("DBG: value=", debug_value)
  # </DEBUG>
  ```

#### 7. 現在実装されているデバッグキー
`main.py` のイベント処理内、`# <DEBUG 2026-07-17>` 〜 `# </DEBUG>` で囲まれた範囲（`STATE_PLAYING`中のみ有効）：

| キー | 動作 |
|---|---|
| F1 | ゴール手前1000ユニットまでワープ |
| F2 | 次のステージへ強制遷移 |
| F3 | 1つ前のステージへ戻る（同ステージ再生成） |
| U | 地面レイヤーの描画オフセットを+1 |
| J | 地面レイヤーの描画オフセットを-1 |

- Stable版へコミットする際も、動作確認・調整用に残す前提（削除する場合はユーザー指示があってから）。
- `STATE_PLAYING`以外では無効（[2026-07-17]F2/F3にガードを追加。以前はGAME_CLEAR画面でF2を押すとランキングが重複登録される不具合があった）。
- ※ `0`キー（エンジン音ミュート切り替え）はデバッグ用ではなく、ユーザー向けの正式機能。
