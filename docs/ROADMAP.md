### 注意事項

#### 1. 新規実装について
- バージョン管理はgitのコミット履歴で行う（フォルダ単位の管理は廃止済み）。
- 新規実装は `main.py` / `src/` を直接編集して進めてよい。
- **実験・試行はブランチで行う**（例: `experiment/scroll-tuning`）。`main` へ直接コミットしない。
- **コミットは AI の判断で行ってよい**（ユーザーの承認は不要）。ブランチ上のコミットはいつでも
  やり直せるため、細かく刻んでよい。ただし下記「コミット前の確認」は必ず満たすこと。
- **`main` へのマージは必ずユーザーの確認を仰ぐ**。マージがレビューの関門であり、ここだけは
  AI の判断で進めない。
- **GitHubへのpushは AI から自動で行わない**。必ずユーザー判断を待つ（外部に出す操作のため）。
- ROADMAP の各課題 [ ] は一つずつ順番に処理する。方針が分岐する場面ではユーザーに確認する
  （実装の細部まで逐一確認する必要はない）。

#### 1-2. コミット前の確認
コミット前に以下を満たすこと。満たせない場合はコミットせず、状況を報告する：

- `python -m pytest tests/ -q` が通ること。
- 挙動を変えた場合は、その変更を実際に動かして確かめたこと（テストだけで済ませない）。
  実機での見た目の確認が必要な場合は、その旨を伝えてユーザーの確認を待ってからコミットする。
- コミットメッセージには「なぜそうしたか」を残す。コードを読めば分かることではなく、
  **検討して採らなかった案とその理由**、**その値でなければならない制約**など、後から復元
  できない情報を書く。
- 検証をしていない場合・失敗した場合は、そのことをメッセージと報告に明記する。

#### 2. バージョンが変わるときに実施する作業（バージョン管理）
**ver名を上げるときだけ**実施する（通常のコミットでは不要）。実験ブランチで刻んでいる段階では
保留し、`main` へマージする区切りでまとめて行う：

1. **ver名の定義**
   - [src/version.py](../src/version.py) の `VERSION` を書き換える。**コード側はここが唯一の情報源**で、
     ウィンドウタイトルは `set_caption(f"Race Game v{VERSION}")` で自動追従する。他のコードに
     ver名を書かないこと。
   - 日付は書かない（gitが持っている情報の手書き複製は必ず陳腐化するため）。

2. **ROADMAP**
   - 更新点を1行で追記する。
   - 形式例：「YYYY-MM-DD　ver名　更新内容」
   - ROADMAPに重複箇所があればその部分を削除

3. **README（公開ページ）**
   - 冒頭のVersionバッジの数字を ver名に合わせる（外部サービスの画像URLのため、ここだけは
     手更新が残る）。
   - 機能追加時は Features の記述も見直す（**ステージを増やしたらステージ数を必ず更新すること**。
     v1.1でStage6を追加した際、READMEが「5 stages」のまま公開されていた実績あり）。

4. **gitタグ**
   - ver名を上げたら `git tag v1.2` のようにタグを付ける（フォルダ名管理をしなくてもその時点へ戻れる）。

#### 2-2. アセットの管理
- `asset/`（画像・音声）は **Git LFS** で管理する（[.gitattributes](../.gitattributes) で png/wav/mp3 を追跡）。
  以前は `asset/` 全体を `.gitignore` で除外していたが、それだとクローンしただけでは
  ゲームが起動しない（`pygame.image.load('asset/bg1.png')` で落ちる）ため撤回した。
- アセットを追加・変更したら普通に `git add` すればよい（LFSが自動でポインタに置き換える）。
- **第三者素材を入れる前に再配布の可否を確認すること**。「作品での使用が自由」と「ファイル
  そのものを公開リポジトリで再配布してよい」は別の許諾で、後者を禁じている配布元は多い。
  **一度pushしたら実質的に取り消せない**（下記の未解決事項を参照）。
  （2026-07-17: BGMを d-elf.com の素材から作者がSunoで作成したものへ差し替え、この懸念は解消）
- **エンジン音（`asset/engine.wav`）は権利関係が未確認のため除外している**（`.gitignore`）。
  無くてもゲームは動く（`sound.py` が `enabled=False` にする）。
  （2026-07-18: エンジン音をコード生成の完全自作品 `engine_low/mid/high.wav` に差し替え、
  この懸念も解消。生成元は `project/engine_sound_gen/generate_engine.py`。`sound.py` は
  旧「1本のWAVをDSP加工＋ピッチシフトして3ループ合成」方式から「3本のループWAVを
  直接ロード」方式に変更した。旧 `engine.wav` は未使用になったが除外設定は維持）

### [x] ① Stage2・3 の水平線右端に、平均を超えて目立つ光の点が出る

**2026-07-18 解決（テクスチャ側を修正）**
- 原因確定: 犯人は `mips[-1]`（縦全平均）ではなく、帯上端が実際に使う**部分ミップ（level4〜7）**。
  `bg2v/bg3v` のテクスチャ下1/3・横91〜96% に焼き込まれた明るい塊（夕焼け/街灯、luma140〜209）が
  部分平均では消えず、`player_z` のテクスチャ1周期(6000)ごとにどのミップ行が来るかで右端(画面x≈745)へ
  明滅していた。ROADMAP当初の「`mips[-1]`を可視化」では左寄りの列しか出ず、この方法では確認できなかった。
- 対処: 地面テクスチャの地面領域だけに、輝度連動スムーズランプで**極端なハイライトのみ上限クリップ**
  （下地の模様は保持）。ミップ側を弄る案（上端を全平均へ寄せる）はスクロール停止・帯の平坦化を招くため不採用。
- 効果: 右端明滅振幅 Stage2 101.8→32.7 / Stage3 96.7→20.0（地面本来の変動幅内）、地面平均輝度 -0.7%（雰囲気不変）。
- Stage4(bg4v)も同位置(x95%,y88%,luma214)に同種の塊があり同じ手法を適用（明滅42.2→18.7）。
  砂は明るいシーンなのでパラメータは別値: CAP=175/ランプ184〜202（暗い地面のStage2/3はCAP=60。
  同じ値だと砂全体が暗くなるため）。平均輝度-0.25%で砂の雰囲気は不変。

- **症状**: スタート直後〜DIST 5937m の間に数回、水平線近くの右端に輝く点が現れる。
  Stage2（夕焼け）とStage3（夜街）で発生。実際の背景の光の部分との落差に見える。
- **分かっている事実**:
  - DIST 5937m は `player_z = 6300`。**地面テクスチャの縦の繰り返し周期は 6000 ワールド単位**
    （テクスチャ高400px ÷ 密度0.0667）なので、報告された範囲はほぼ**テクスチャ1周期ぶんと一致**する。
    → 6000ごとに繰り返している可能性が高い。「5937mで止まった」のか「その先も出続けるが
    気づかなかっただけ」なのかで原因の範囲が変わるため、まずそこを確認したい。
  - 水平線付近はミップのレベル5〜7が効き、テクスチャを縦に67〜267テクセル平均している。
  - 帯の上端は `scale` が画面被覆の下限にクランプされ、横に引き伸ばされている
    （Stage2で y=317〜365、Stage3で y=317〜371）。
- **仮説（未検証）**: `asset/bg2v.png` / `bg3v.png` に含まれる強い光（夕焼け・街灯）が、
  縦方向のミップ平均では消えずに「明るい列」として残り、上端の横引き伸ばしで幅広の光点として
  見えている。ミップは縦にしか平均しないため、横方向の高輝度は保存される。
- **確認方法**: `GroundLayer.mips[-1]`（1px高＝真の縦平均）を全列ぶん可視化し、突出して明るい
  列があるか見る。あれば原因はテクスチャ側。`player_z` を6000ずつ進めて再現するかも確認する。

### [x] ② 直進中、水平線と車体の中間あたりで地面が左右から狭まって見えることがある

**2026-07-18 解決（吸い寄せ感ほぼゼロを実機確認）**
- 原因確定: **横方向の時間エイリアシング**。測定では粗い模様は毎フレーム5〜10pxで正しく
  外向きに流れており（遠近の拡大流）、細かい模様だけがナイキスト限界（周期<2×流速）を
  超えて逆走・内向きに滑る。この混在フローが「中央へ吸い寄せられる」錯視の正体。
- 当初有力だった縦方向のwagon-wheel説は**縦モーションブラー（速度連動ミップ下限）を
  実装して棄却**: 高周波の逆走は測定上消えたが実機の吸い寄せ感は不変で、かつユーザー
  評価はブラー無しの旧版が良い、だった。横周波数は縦ミップでは落ちないため道理。
- 対処（ブラー無しの2点）:
  1. 縮小ストリップの横スケールを nearest → **smoothscale（面積平均）** に変更。
     ニアレスト縮小は列の間引きで、フレーム毎に違う列が落ちて偽の細かいチラつき
     （エイリアシング源）を量産していた。拡大側（近景）はニアレストのままでくっきり感維持。
  2. `GROUND_TEXTURE_WORLD_WIDTH` を 24000 → **41800**（上記「確認方法」の対処案(a)）。
     模様が1.74倍大きくなりエイリアシング域のエネルギーが減少。テクセル前進も
     10.8→6.2/frameに低下。現実装は縦横とも同一定数から決まるため41800でも等方のまま。
- デバッグキー`M`で新旧を実機A/B比較できるようにした（下記デバッグキー表）。
- 注意: テクスチャの縦繰り返し周期は 6000 → **約10450ワールド単位** に変わった
  （①の記述にある6000は当時の値）。

- **症状**: 直進しているのに、地面が左右から距離を詰めてくるように見えるときがある。
  本来は消失点から手前に向かって幅が拡大する（逆方向）はず。
- **分かっている事実**:
  - **符号の反転ではない**。直進時は `shift≈0` で `scale = max(a*(offset+dy), 0.501)` となり、
    `dy` に対して単調非減少。実装上、幅が縮むことはない。
  - よって見た目の逆行であり、**時間方向のエイリアシング（wagon-wheel効果）が最有力**。
    ミップマップは空間方向のエイリアシングしか直さないので、これは残る。
  - 中間距離（dy≈150）はミップlevel 0（元テクスチャそのまま）が使われ、フィルタが一切かからない。
  - 最高速では1フレームあたり **10.8テクセル** 進む（162 × 密度0.0667）。テクスチャに周期
    21テクセル未満の模様があるとナイキスト限界を超え、逆行して見える。
  - **改修で悪化した可能性が高い**: 密度を 0.0383 → 0.0667 に上げた（等方にした）ため、
    1フレームの進みが 6.2 → 10.8 テクセルへ**1.74倍**になった。
- **確認方法**: `GROUND_TEXTURE_WORLD_WIDTH`（[background.py](../src/background.py)）を
  24000 → 41800 に戻すと密度が旧実装相当になる。これで症状が消えるなら密度が原因と確定できる。
  その場合の対処案は (a) 密度を下げる（模様は粗くなる）、(b) 速度に応じてミップレベルを
  上げ気味にする（動いているときだけボカす）、(c) テクスチャ側の模様の周期を見直す。

### [x] ③ 縦スクロールに合わせて周期的な色むらパターンが流れてくる

**2026-07-18 解決（ミップ仮説は棄却、テクスチャ側の行平均を修正）**
- 原因確認: ユーザーの見立て（ミップの縦方向の問題）は**検証の結果棄却**。PILの正確な
  面積平均で、各ミップlevelの行平均をlevel0のブロック平均と比較したところ最大差1.6
  （8bit丸め誤差程度）で、ミップは行平均を忠実に保存していた。
- 真因はテクスチャ自体：(1) 行平均輝度のゆっくりした低周波の波（p2p 4〜15、RGBチャネル
  ごとに独立、bg1v/3vはG成分の波が特に大きい）、(2) **縦ループの継ぎ目（最終行→先頭行）
  の行平均段差**（p2p 18〜20、5枚中4枚）。1周期＝約10450ワールド単位（②の対策で
  6000から伸びた）ごとにこれが降ってくるのが「周期的な色むら」の正体。
- 対処: 全5枚の地面テクスチャに前処理を適用。(1)行平均の低周波成分をRGBチャネル別に
  全体平均へ寄せる補正（細かい模様・行内の変化は保持）、(2)継ぎ目前後40行に線形ランプで
  段差を分散。①のハイライトクリップ（bg2v/3v/4v）は均した後に同パラメータで再適用し干渉を防止。
- 効果: 画面帯の1周期輝度波 13〜21→2〜14、継ぎ目段差 18〜20→0〜8。①の右端明滅は回帰なし
  （bg2v 31.8→37.4は測定サンプル数の違いによる誤差、実効的な悪化ではないことを実機で確認）。
  実機でむらの軽減を確認済み。
- 残課題: bg4v(砂)は残存14とやや大きい。行平均では消せない横方向の大きな砂パッチが原因。
  気になる場合は縦横2D低周波均しへ拡張する余地あり（今回は許容範囲としてスコープ外）。
